Run an interactive shell command and let the user answer prompts mid-execution.

Use this when a CLI is going to ask the user for something at runtime — auth flows that need a one-time code, device-login code, token, password, `gh auth login`, `npm login`, `vercel login`, `aws sso login`, package installers that prompt y/N, or any tool that pauses waiting on stdin.

How it works:
- Spawns the command in a PTY (so CLIs that only prompt when stdin is a TTY work correctly).
- Streams output live so the renderer shows the running terminal as it happens.
- Watches for prompt patterns you declare. When a prompt matches, the agent either writes a known `answer` directly into the PTY, or asks the user with the standard question dock and then writes the user's answer into the PTY.
- OAuth flows that print an `/oauth/authorize?...code=true...` URL automatically show a code-capable browser sign-in question with the OAuth URL. If the browser displays a code, the user's answer is written into the terminal even if the CLI prints its paste prompt later. If the browser completes automatically, the question closes when the terminal exits.
- If a CLI prints an obvious stdin prompt for a code, token, OTP, passcode, or password and no declared prompt matched, the tool automatically asks the user and writes the answer into the terminal. This covers Claude variants that show a browser code and then print `Paste code here...` without the agent having declared a pattern.
- If a CLI prints a plain `Press Enter to retry/continue` prompt after a failed auth attempt, the tool sends Enter automatically so the flow can return to the next code prompt instead of freezing.
- The user never types directly into the terminal. Terminal input is controlled by the agent/tool.

Required prep:
1. Probe the command FIRST with the regular `bash` tool (or read the CLI's `--help` / docs) to learn what prompts it actually prints — verbatim. Don't guess.
2. Call `bash_interactive` with `prompts` covering each prompt you saw. The `pattern` is a JS regex matched case-insensitively against new PTY output; pick something distinctive enough to not match surrounding chatter. You do not need to declare a code prompt for Claude-style OAuth URLs; the tool handles that fallback automatically.
3. For choices the agent already knows, set `answer`. Use an empty string to press Enter/select the CLI default. For choices the agent does not know, omit `answer` and phrase `question` for the user.

Each prompt fires AT MOST ONCE per entry. If the same prompt can appear twice, include the entry twice.

Differences from `bash`:
- PTY-backed (real TTY).
- Streams output as it happens via the message metadata channel — the renderer shows the running terminal live.
- Mid-command input via agent-known `answer` values or user answers collected through the standard question dock.

Parameters:
- command: the shell command line to run.
- prompts: array of { pattern, question, header?, answer? }. Required for any flow that pauses for input.
- timeout: ms to allow the command to run (default 120000, max 600000).
- cwd: working directory.
- description: short description shown in the UI.

Returns the full captured output (stdout + stderr interleaved as the PTY produced it) and the exit code in metadata. The tool aborts and reports an error if the command runs past `timeout` or the user dismisses a prompt.

Don't use this for non-interactive commands — use `bash` instead. Don't use this for long-running servers — there's no streaming back to the model after exit.
