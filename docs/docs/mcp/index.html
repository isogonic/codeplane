MCP servers

Plug any Model Context Protocol server in as a tool source. Codeplane spawns the process, pipes JSON-RPC over stdio, surfaces tools / resources / prompts to the agent.

How MCP fits in

The agent's native tools (file edit, shell, search, git) cover most coding flows. MCP servers add capabilities Codeplane doesn't ship — GitHub issue management, Linear / Jira sync, browser automation, custom internal tooling. The model decides which tool to call; you don't have to swap modes.

Adding a server

In any codeplane.json (user or project):

"mcp": {
  "filesystem": {
    "command": "npx",
    "args":    ["-y", "@modelcontextprotocol/server-filesystem", "/Users/me/Code"]
  },
  "github": {
    "command": "uvx",
    "args":    ["mcp-server-github"],
    "env":     { "GITHUB_PERSONAL_ACCESS_TOKEN": "{env:GITHUB_TOKEN}" }
  },
  "sequentialthinking": {
    "command": "npx",
    "args":    ["-y", "@modelcontextprotocol/server-sequential-thinking"]
  }
}

Codeplane spawns each entry on session start, then listens for tool / resource / prompt advertisements over stdio. The MCP standard handles versioning + capability negotiation — Codeplane keeps the picky parts to itself.

Per-key reference

KeyRequiredWhat it does
commandyesExecutable to run (looked up in $PATH).
argsnoString array passed to the command.
envnoExtra environment variables — supports {env:VAR} placeholders.
cwdnoWorking directory for the process. Defaults to the project root.
enablednoSet false to temporarily disable without removing the entry.
timeoutnoPer-tool-call timeout in milliseconds (default 30000).

Permissions for MCP tools

Every tool exposed by an MCP server lands in the same permission flow as native tools. The first call from a new tool triggers an Allow / Deny / Always prompt in the active surface; the choice is persisted under the session (or directory) scope. Bulk-allow via codeplane.json:

"permission": {
  "tool": {
    "mcp__github__create_issue": "allow",
    "mcp__github__*":            "ask",
    "mcp__filesystem__*":        "allow"
  }
}

The mcp__<server>__<tool> naming maps directly to what the MCP spec advertises. Globs match suffixes per segment.

Debugging

Run the server with --log-level DEBUG and look for lines tagged mcp.<name> — they include the JSON-RPC frames sent and received. If a server isn't showing up:

  1. Check the command exists in $PATH for the user the server runs as.
  2. Make sure stdout isn't being polluted — MCP servers must reserve stdout for protocol traffic and log everything else to stderr.
  3. Set "enabled": false on every other entry to isolate the broken one.

Writing your own

An MCP server is a small binary that speaks JSON-RPC over stdio. The official SDKs (@modelcontextprotocol/sdk for TS, mcp for Python, mcp-go for Go) handle the framing. A 40-line server can expose a useful tool. See modelcontextprotocol.io/quickstart.
