Self-hosting

Run a Codeplane server on a box you own, then point every desktop / mobile / browser at it. One server, one set of sessions, every device.

Minimum requirements

  • Linux x86_64 or arm64 (also runs on macOS / Windows servers, but Linux is the focus here).
  • 1 GB RAM idle; 2-4 GB peak when running large diffs / parsing big trees.
  • Outbound HTTPS to your model provider (OpenAI / Anthropic / OpenRouter / your own).
  • Inbound TCP on whatever port you bind (default 4096).

Install the binary

curl -fsSL https://codeplane.cc/install | bash
# OR: download the platform tarball from /releases/latest manually

Pin a version in production with -v 28.1.20 so an auto-upgrade can't surprise you mid-deploy.

Run as a systemd service

Create /etc/systemd/system/codeplane.service:

[Unit]
Description=Codeplane server
After=network-online.target
Wants=network-online.target

[Service]
Type=simple
User=codeplane
Group=codeplane
EnvironmentFile=/etc/codeplane.env
ExecStart=/usr/local/bin/codeplane serve --port 4096 --hostname 127.0.0.1
Restart=on-failure
RestartSec=3
LimitNOFILE=4096

[Install]
WantedBy=multi-user.target

Drop secrets in /etc/codeplane.env (mode 0600):

ANTHROPIC_API_KEY=sk-ant-...
OPENAI_API_KEY=sk-...
CODEPLANE_HOME=/var/lib/codeplane
CODEPLANE_LOG_LEVEL=INFO

Enable + start:

sudo useradd -r -m -d /var/lib/codeplane codeplane
sudo systemctl daemon-reload
sudo systemctl enable --now codeplane
sudo journalctl -fu codeplane     # follow the logs

Reverse proxy (nginx)

Codeplane speaks plain HTTP + Server-Sent Events. Front it with nginx for TLS:

server {
  listen 443 ssl http2;
  server_name codeplane.example.com;

  ssl_certificate     /etc/letsencrypt/live/codeplane.example.com/fullchain.pem;
  ssl_certificate_key /etc/letsencrypt/live/codeplane.example.com/privkey.pem;

  location / {
    proxy_pass         http://127.0.0.1:4096;
    proxy_http_version 1.1;
    proxy_set_header   Host              $host;
    proxy_set_header   X-Real-IP         $remote_addr;
    proxy_set_header   X-Forwarded-For   $proxy_add_x_forwarded_for;
    proxy_set_header   X-Forwarded-Proto $scheme;

    # SSE: disable buffering + raise timeouts.
    proxy_buffering    off;
    proxy_read_timeout 24h;
    chunked_transfer_encoding off;
  }
}

Reverse proxy (Caddy)

codeplane.example.com {
  reverse_proxy 127.0.0.1:4096 {
    flush_interval -1
  }
}

Caddy issues + renews the Let's Encrypt cert automatically. flush_interval -1 disables buffering so SSE streams in real-time.

Docker

docker run -d --name codeplane \
  -p 4096:4096 \
  -e ANTHROPIC_API_KEY=sk-ant-... \
  -v codeplane-data:/data \
  -e CODEPLANE_HOME=/data \
  ghcr.io/devinoldenburg/codeplane:latest \
  serve --port 4096 --hostname 0.0.0.0

Image is published as ghcr.io/devinoldenburg/codeplane:<version> and :latest. Multi-arch (linux/amd64 + linux/arm64).

Authentication

By default the server has no auth — anyone who can reach the port can drive the agent. Two ways to lock it down:

  1. Bearer token. Pass --auth <token> on the server; clients send Authorization: Bearer <token>. Built into the CLI + desktop + mobile picker.
  2. Reverse proxy auth. Wrap with mTLS, OIDC, Cloudflare Access, Tailscale Funnel, etc. Codeplane trusts the upstream IP / headers — no app-level setup needed.

For exposed deployments behind nginx, the simplest path is Tailscale or Cloudflare Access — both terminate auth before any traffic reaches Codeplane.

Backups

State lives in $CODEPLANE_HOME (default ~/.codeplane; /var/lib/codeplane in the systemd unit above):

  • sessions.db — SQLite, holds every message, tool call, permission record.
  • config/ — merged config + auth tokens.
  • cache/ — model + tool caches; safe to drop, will rebuild.

Stop the service before snapshotting sessions.db for a consistent copy. sqlite3 sessions.db ".backup '/backup/codeplane.db'" works on a running server too.

Upgrading

sudo systemctl stop codeplane
sudo codeplane upgrade               # or pin: codeplane upgrade 28.1.20
sudo systemctl start codeplane
sudo journalctl -fu codeplane        # confirm no migration errors

Migrations run at the next start. They're forward-compatible — rolling back to an earlier minor version after upgrade is safe; rolling back across a major version isn't supported.
